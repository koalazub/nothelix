# Output-duplication fix report

Notebook cell OUTPUT (virtual rows) rendered duplicated at stale anchors. The
store holds ONE copy; the screen showed several. Root cause: the Helix fork
remaps inline-IMAGE positions (`raw_content`, char-keyed) through document
edits, but the `output_lines` virtual-row annotation (line-keyed) was NOT
remapped. On an edit above a cell, output rows stayed pinned to the OLD line
key while the plugin re-rendered at the NEW anchor, so copies accumulated at
multiple line keys.

## Repo 1 — /Users/koalazub/projects/helix (fork remap)

Commit: `867f649e` (jj change qwtrmpqq)
`fix(annotations): remap output_lines through document edits`

- `helix-view/src/annotations/output.rs`: added
  `OutputLines::remap_lines(&mut self, f: impl Fn(usize) -> Option<usize>)`.
  It drains the `below` map and reinserts each entry at `f(key)`, dropping keys
  that map to `None`. Collisions (two old lines mapping to the same new line)
  resolve last-writer-wins; the plugin sets at most one entry per anchor, so a
  collision only arises from a pathological edit and either surviving row set is
  defensible.
- `helix-view/src/document.rs`: in the transaction-apply path, immediately after
  the `raw_content` remap loop, remap `output_lines`. Because keys are LINE
  indices (not char indices) `remap_positions` cannot be reused verbatim; for
  each key I convert the anchor line to its start char in the OLD text
  (`old_doc.line_to_char`), map that char through the `ChangeSet` with
  **`Assoc::After`** (the association every other annotation here uses —
  diagnostics, inlay hints, raw_content, highlights), then convert back with
  `self.text.char_to_line` in the NEW text. Guards: skip keys `>= old_doc
  .len_lines()`, drop results whose char lands past `new_len`.
- Test `remap_lines_shifts_keys_and_drops_none` in the existing `mod tests`:
  rows set at line 5 and line 2; a `line >= 3 -> +2` closure shifts line 5's
  rows to line 7 (asserted) and drops line 2 (mapped to None) — asserts old keys
  are gone and the moved rows carry the right text.

## Repo 2 — /Users/koalazub/projects/nothelix (defense-in-depth)

Commit: (see below)
`fix(output): clear stale output rows before restoring on open`

- `plugin/nothelix/execution.scm`: in `restore-cell-outputs-on-open!`, call
  `(try-clear-all-output-lines!)` once — after `rope`/`total-lines` are computed,
  immediately before the per-cell `for-each` render loop. `try-clear-all-output
  -lines!` is already exported from `output-render.scm` and is `with-handler`-
  wrapped, so it is a safe no-op on an old binary lacking the FFI. The per-run
  `clear-cell-output!` path is untouched.
- Net-neutral: `restore-cell-outputs-on-open!` re-renders EVERY stored cell in
  the very next loop, so clear-all-then-full-rerender adds exactly one fresh copy
  per cell. The only rows it drops are stale duplicates keyed to now-defunct
  line indices — which is precisely the bug. No output the user expects to keep
  is lost.

## @show-vs-repr (report only — NOT a bug)

Confirmed from
`~/.local/share/nothelix/outputs/…_kata_jl/cell-3.json`. The stored JSON has TWO
distinct kernel outputs: a `stream`/stdout carrying the `@show` lines
(`eigvals(A) = …`, `tr(A) = 5`, `det(A) = -25.0…`) and a separate
`execute_result` with `text/plain: -25.0…` (the bare return value). The
`###NOTHELIX-OUTPUT-ROWS###` section contains BOTH — the three @show lines then
the bare repr. This is genuine dual capture matching Jupyter semantics for a
cell that does `@show det(A)` then a bare `det(A)`; it is NOT a plugin
double-append and nothing here needs fixing.
