system architecture
  1. 01 Forked Helix · the host editor core · RawContent annotation layer · math-line rows · Steel FFI surfaces ▸
    Renders the buffer and owns the one idea the core had to grow: a plugin can attach raw bytes at a document position, and those bytes consume visual rows. Images are drawn through Kitty’s Unicode-placeholder protocol so they survive scrolling and edits. Everything above this is policy the plugin supplies.
    Rusthelix-viewhelix-tui
  3. 02 plugin/ · Steel / Scheme commands · keymaps · pickers · overlay + image placement · lifecycle hooks ▸
    Everything editor-shaped. The :commands, the keymaps, the cell and chart pickers, where overlays and images land, and the document-lifecycle hooks that conceal math, reserve space for renders, and refresh on save. It is the caller and the UI; the heavy lifting is delegated downward.
    SteelScheme
  5. 03 libnothelix · Rust cdylib notebook parsing · kernel IPC · Kitty encoding · Typst + MiTeX · error enrichment ▸
    The system work, kept off the editor thread. .ipynb↔.jl conversion, kernel process management and file IPC, image decode and Kitty encoding, the LaTeX→Typst math pipeline, and the enrichers that turn a bare stack trace into something that names your variables.
    Rustserdetypstmitex
  7. 04 Julia kernel · one per document persistent module · cell registry · output capture · type introspection ▸
    A long-lived process that runs each cell in a persistent module, so definitions accumulate exactly like a REPL session. It tracks the cell registry and the type of every binding, captures stdout and figures, and emits structured JSON for both results and errors.
    JuliaJSON3
▸Hover or click any layer to read what it does. Replay walks the four layers, then traces one cell run through them.
